## (a) 題目解剖

- **開頭元素＝願景/應用場景**，不是案例、不是機制、也不是方法：主標題 "Towards digitalized maintenance of operating tunnels" 以「邁向式（Towards + 動名詞願景）」領頭，賣的是**營運隧道維養數位化**這個產業級目標。案例城市/路線完全不入題。
- **方法詞全部押在副標題**：`text documents-based`（資料源限定詞）＋ `defect evaluation and visualization`（兩個交付物名詞）。沒有演算法專名（fuzzy rough set、text classification 都留在摘要），方法詞只到「資料型態＋動作」層級——與 Zhang 2025 筆記發現的「題目只放方法大類、專名留摘要」同一套路。
- **構式**：`Towards [願景名詞短語]: A [資料源]-based [交付物A] and [交付物B]`。冒號前賣願景（入口寬、蹭數位轉型熱詞），冒號後給差異化錨點（別人用影像/點雲，我用**文字報告**）。
- 隱含策略：整個賽道都在做 vision-based 檢測，把 "text documents-based" 放進題目本身就是缺口宣言——**題目即缺口**。

## (c) 前言手法（依參考文獻叢集反推；段數未知）

63 篇參考文獻的叢集結構清晰暴露前言的 staging 路線：

| 叢集 | 篇數概估 | 前言角色【推測】 |
|---|---|---|
| 影像/深度學習病害檢測（crack/leakage 分割、Mask R-CNN、高速取像） | ~15+ | 「自動檢測已成熟」的鋪墊層 |
| GPR/LiDAR/點雲/機器人等感測 | ~8 | 同上，感測面向 |
| 隧道狀態/風險評估（fuzzy、Bayesian、MCDA、resilience） | ~10 | 「評估方法存在但主觀/單屬性」的限制層 |
| Fuzzy rough set 方法論（Info. Sci. 2022、Appl. Intell. 2023、IEEE TFS） | ~4 | 方法正當性來源（借數學社群的成熟工具） |
| 文本/本體論/BIM 資訊萃取（ontology、building codes、ISARC） | ~5 | 「文字資料可被機器處理」的可行性層 |
| 數位孿生/O&M 決策框架 | ~4 | 願景層，呼應題目 Towards |
| TUST 自刊引用 | ~17 | 期刊對位（近 1/3 落在 TUST） |

- **缺口 staging【推測】**：三層漏斗——(i) 感測與影像檢測已大量自動化；(ii) 但數十年累積的**人工巡檢文字報告**是沉睡資產，幾乎無人結構化利用；(iii) 既有評估法未處理多屬性間的關聯與模糊性。缺口本質是「**資料源缺口＋方法移植**」：把 NLP 文字分類與 fuzzy rough set 從資訊科學搬進隧道維養。
- **貢獻宣告句式**（據公開摘要轉述）：採「not only… but also…」雙賣點——不僅自動化萃取報告中的病害資訊，還能考慮多屬性關聯來評定嚴重度。與 Zhang 2025 完全相同的句型，可視為 TUST 高頻貢獻句式。
- 摘要自我定位詞是 "practical framework"——不主張理論突破，主張**工程可用性**，用 10,345 筆真實病害背書。
- 文獻年代高度集中 2019–2024（63 篇中約 50 篇），前言是「貼著當下熱度寫」的typical打法。

## (f) 缺陷包裝【推測為主】

- 摘要完全不提限制——標準操作。
- 從摘要即可預見、但幾乎肯定被淡化的弱點：
  1. **文字報告的品質依賴**：巡檢報告的用語不一致、漏記、主觀描述，直接決定分類上限；
  2. **無結構力學內涵**：嚴重度評級來自屬性關聯統計，不接裂縫力學機制、不接安全係數；
  3. **語言與範式移植性**：以中文巡檢報告訓練的分類器移植到他國體系的效度；
  4. 病害「演化」（時間軸）是否處理？摘要隻字未提，很可能是靜態快照評估。
- 這些若在文中處理，慣用包裝法【推測】是 future work 條款（接影像資料、接 BIM/數位孿生、擴充病害類型）——把缺陷轉寫成路線圖。

## (h) 對我們的啟示（營運山岳鐵路隧道襯砌裂縫 × 地下水位循環 × 跨尺度 FDM-DEM）

1. **「operating tunnels」是 TUST 的熱鍵詞，且願景式主標題被市場驗證**：本文靠「Towards＋營運隧道維養」的題目在 18 個月吸引 22 次引用。我們的題目應讓「營運/在役（operating/in-service）隧道」出現在主標題位置，把 FDM-DEM 等方法專名壓到副標題甚至只留摘要——與 Zhang 2025 筆記結論互相印證，這已是兩篇樣本一致的 TUST 題目公式。
2. **它是我們天然的「互補缺口」引文**：Ou 的評估純靠文字報告統計屬性關聯，**沒有任何力學機制**。前言可用它代表「維養評估已數位化、但嚴重度評級缺乏機制依據」這一層，順勢推出我們的貢獻——地下水位循環驅動的裂縫演化機制給嚴重度評級提供物理基礎。數位維養框架（Ou）與機制模型（我們）是上下游關係，引用時把自己放在「供給機制內核」的位置。
3. **規模數字進摘要**：本文把 10,345 筆病害寫進摘要當主錨點。我們對應的動作：把監測年數、水位循環次數、裂縫條數、模型單元/顆粒數量級等**具體大數**寫進摘要，framework 賣規模、機制文賣「現場數據×模擬尺度」的組合拳。
4. **參考文獻的期刊對位策略**：63 篇中約 17 篇 TUST（~27%），且各模組都有對應文獻叢集支撐。我們投 TUST 時各支柱（襯砌裂縫、地下水—隧道互制、跨尺度離散連續耦合）都應配足近五年 TUST 引文，總體 TUST 佔比拉到 1/4 以上——目前主庫 REFS_MASTER 的 TUST 55% 已超標，反而要注意別過度單一。
